Understanding package managers by comparing npm and pip

Nitish
 2023-05-09  2023-05-09  1125 words   6 minutes 

Quick intro maybe..

A package manager is a software tool that automates the process of installing and managing software dependencies. Common examples includes npm for Node.js, pip for Python, and apt for Debian-based Linux distributions.

Dive in a bit more?

  • While different package managers differ in terms of syntax and configurations, they pretty much follow the same workflow:

  1. Define dependencies: Developers define the dependencies required for their project in a configuration file.

  2. Install dependencies: The package manager automatically downloads and installs the required dependencies, including any additional packages needed to satisfy dependencies.

  3. Manage dependencies: Developers can use the package manager to update, remove, or add new dependencies to their project.

  4. Dependency resolution: The package manager ensures that all dependencies are compatible with each other and are installed in a specific order to avoid any conflicts.

Choosing the appropriate package manager for your project is crucial as it plays an essential role in modern software development, which can significantly impact productivity and efficiency.

Okay, I’ve been using them for long, and I understand it. What’s more to notice?

  • We’ll talk about npm for Node and pip for Python here and limit our discussion to these two. Through comparison, we’ll understand some of the subtle things we may have overlooked in the past:
  1. How do we define dependencies in both (configuration):
  • Node user package.json file, while Python uses requuirements.txt file: Basically, configuration data format could be anything specific to the ecosystem(think of other languages/formats: xml, etc. )
  1. Dependency resolution

  • In Node.js, packages are versioned using Semantic Versioning, or SemVer for short. SemVer defines a set of rules for how version numbers should be assigned and incremented, based on a three-part numbering system: major.minor.patch. The major version number should be incremented when there are significant changes that may break compatibility with older versions, the minor version number should be incremented when new features are added in a backward-compatible manner, and the patch version number should be incremented for bug fixes and other minor changes.

  • When specifying dependencies in a Node.js project, you can specify exact versions using the package.json file, which lists all the dependencies of the project along with their version numbers. This file can be committed to version control, ensuring that every developer working on the project uses the same set of dependencies.

  • To update a package in a Node.js project, you can use the npm update command. This will update all the packages in the project to their latest compatible versions, as determined by the version ranges specified in package.json. If you want to update a specific package to a newer version, you can run npm update package_name.

  • Before updating, it’s a good practice to “freeze” the versions of all dependencies, which locks the versions to the specific ones listed in package-lock.json. This file is automatically generated by npm and specifies the exact versions of all dependencies, including their transitive dependencies. It should be committed to version control along with package.json to ensure that everyone working on the project uses the same set of dependencies.

  • In Python, packages can be versioned using either exact version numbers or version ranges. To specify an exact version number for a package, you can use the package_name==version syntax in a requirements.txt file, which lists all the project’s dependencies along with their version numbers. To specify a range of acceptable versions, you can use operators such as >= or <=.

  • To update a package in a Python project, you can use the pip install --upgrade command, followed by the name of the package you want to update. This will update the specified package to the latest version compatible with the version range specified in requirements.txt. To update all packages in the project, you can use the pip freeze > requirements.txt command to generate an updated requirements.txt file, which specifies the latest compatible versions of all dependencies.

  • Before updating, it’s a good practice to “freeze” the versions of all dependencies by running pip freeze > requirements.txt. This generates a requirements.txt file that lists all the project’s dependencies along with their exact version numbers. This file can be committed to version control to ensure that everyone working on the project uses the same set of dependencies.

To summarize, in order to update packages in both Node.js and Python projects, it is crucial to “freeze” the versions of all dependencies by generating a lock file (i.e., package-lock.json for Node.js or requirements.txt for Python) that lists the exact versions of all dependencies. These lock files should be committed to version control along with the corresponding package.json or setup.py file, respectively, to ensure that everyone working on the project uses the same set of dependencies.

  1. Dependency Isolation :
  • The goal is to manage dependencies in a software project in a way that ensures they are separate and isolated from each other, as well as from the system’s dependencies. This helps to prevent conflicts between dependencies, and ensures that the project can be easily reproduced on different systems.

In practical terms, dependency isolation involves creating a separate environment for each project, with its own set of dependencies. This is achieved using tools like package managers or virtual environments, which create a sandboxed environment in which the project can run without affecting other projects or the system as a whole.

  • In Node.js, package-lock.json is used to lock down exact package versions and prevent updates to dependencies without explicit approval. This ensures that all developers on a project are working with the same set of dependencies. In addition, Node.js supports multiple versions of the same package by using separate peer dependencies. This allows projects to use different versions of a package without causing conflicts.

  • Python, on the other hand, uses virtual environments to isolate project dependencies from the system’s Python installation and other projects. This allows multiple projects with different dependencies to run on the same system. Python also supports multiple versions of the same package by installing each version into its own virtual environment.

  • Despite the measures taken to ensure dependency isolation, conflicts can still arise between dependencies due to peer dependencies. Node.js handles these conflicts by prioritizing the version specified by the direct dependency over the peer dependency. Python, on the other hand, handles these conflicts by using a package resolution algorithm that takes into account the version specified by the direct dependency as well as the requirements of other dependencies.

It’s important to follow best practices such as freezing the versions of all dependencies, keeping dependencies up to date, and using tools like npm-check or pip-tools to manage dependencies.
